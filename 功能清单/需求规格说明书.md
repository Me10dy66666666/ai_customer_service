# AI智能客服系统需求规格说明书（MVP版本）

## 1. 简介

### 1.1 文档目的
本文档基于MVP实际实现情况，更新AI智能客服系统的功能需求、非功能需求、数据需求和验收标准，确保文档与实际系统一致，为后续迭代开发提供准确依据。

### 1.2 术语定义
| 术语 | 解释 |
|------|------|
| AI智能客服 | 基于人工智能技术的自动化客户服务系统，能够理解用户意图并提供相应服务 |
| 知识库 | 存储产品信息、常见问题等数据的系统，为AI客服提供知识支撑 |
| 意图识别 | 识别用户提问或需求的真实意图的技术 |
| 多轮对话 | 保持上下文连贯性的连续对话能力 |
| 工单 | 需要人工处理的客户问题记录 |
| 用户画像 | 基于用户历史数据构建的用户特征模型 |
| 转化率 | 从咨询到下单的转化比例 |
| 游客会话 | 未注册用户的会话，通过sessionId进行标识和跟踪 |

## 2. 项目概述

### 2.1 项目背景
随着企业业务的快速发展，传统客服模式面临着人工成本高、响应速度慢、服务质量参差不齐等挑战。AI智能客服系统通过整合人工智能技术，实现自动化客户服务，提高服务效率和质量，同时降低运营成本。

### 2.2 项目目标
- 实现售前产品咨询的AI解决率≥80%
- 实现售后产品问题的AI解决率≥70%
- 降低售前产品咨询的人工占比至少70%
- 提升引导用户下单的转化率8%~12%
- 提供7×24小时不间断的智能服务
- 支持50+用户并发（MVP阶段）

### 2.3 目标用户
- **游客用户**：未注册用户，可通过会话ID进行标识，仅限基础咨询（产品/价格）
- **注册用户**：通过AI智能客服获取深度产品咨询和方案匹配的用户
- **会员用户**：享受专属服务的注册用户（userType=2）
- **系统管理员**：管理知识库、查看数据分析、处理工单等（userType=3）

## 5. 数据需求

### 5.1 数据实体
| 实体 | 描述 | 属性 |
|------|------|------|
| 用户 | 系统用户，包括注册用户、会员用户和管理员 | 用户ID、用户名、密码、手机号、邮箱、user_type(1-普通,2-会员,3-管理员)、状态 |
| 游客会话 | 未注册用户的会话记录 | session_id、user_type(0-游客)、user_input、ai_response、dify_conversation_id |
| 会话 | 用户与AI客服的对话记录 | 会话ID、用户ID/session_id、开始时间、结束时间、会话内容、解决状态 |
| 知识库 | 存储产品信息和常见问题的数据 | 知识ID、标题、内容、分类、文件类型、dify_document_id、状态 |
| 工单 | 需要人工处理的客户问题 | 工单ID、用户ID、问题描述、创建时间、状态、优先级、处理人、处理结果 |
| 产品订单 | 用户历史订单信息 | 订单ID、用户ID、订单号、产品名称、型号、价格、总金额、状态 |

### 5.2 用户类型定义（实际实现）
- **游客用户**: user_type = 0，通过 consultation_logs.session_id 标识，userId = null
- **注册用户**: user_type = 1，通过 users 表存储，需要 JWT 认证
- **会员用户**: user_type = 2，通过 users 表存储，需要 JWT 认证  
- **系统管理员**: user_type = 3，通过 users 表存储，需要 JWT 认证

## 6. 范围限定

### 6.1 功能范围（MVP阶段）
- **已实现功能**: 网页端AI客服、知识库管理、用户注册登录、工单系统、历史订单同步、异常处理机制
- **未实现功能**: 微信服务号接入、用户画像自动构建、复杂多轮对话管理、促销规则解析
- **技术集成**: 使用 Dify 作为 RAG 和 LLM 服务，MySQL 作为数据库，Spring Boot + Vue.js 作为技术栈

### 6.2 技术范围
- 系统采用本地开发环境部署
- 集成 Dify AI 平台提供 RAG 和对话能力
- 前端使用 Vue.js 3 + Vite，后端使用 Spring Boot 3 + MyBatis（Mapper + XML）
- 持久层统一规范：仅使用 Mapper + XML，不新增 JPA Repository
- 查询规范：单记录查询使用 LIMIT 1，存在性判断统一使用 EXISTS
- 服务层查询规范：禁止全量查询后在内存中过滤，筛选条件下推到 Mapper SQL

## 8. 风险/依赖备注

### 8.1 风险因素
| 风险 | 影响 | 应对措施 |
|------|------|----------|
| Dify API稳定性 | 影响AI客服响应质量和可用性 | 实现重试机制和降级策略，配置合理的超时时间 |
| 知识库内容质量 | 直接影响AI客服的服务质量 | 建立完善的知识库管理流程，定期更新和补充内容 |
| 游客会话管理 | 影响未注册用户的体验一致性 | 优化sessionId生成和存储机制，确保会话连续性 |

### 8.2 依赖关系
| 依赖 | 说明 | 影响 |
